# OpenChat 下一轮执行计划 v11

日期: 2026-04-06
范围: `apps/openchat`
当前阶段: 生命周期收口、UUID 隔离和标准 E2E 执行链路已闭环，下一轮进入“测试洁净度 + 消息/会话一致性深化 + 持续架构审计”

## P0. 清理测试噪音并提升失败可诊断性

目标:

1. 收敛负向单测中的预期 error/warn 日志
2. 避免测试输出被大量已知噪音淹没

建议动作:

1. 审视 `message.service.spec.ts`、`ws.gateway.spec.ts` 等故意制造失败路径的单测
2. 对明确预期的异常路径用 logger spy 或更窄的 mock 抑制噪音
3. 保留真正异常时需要看到的日志

退出标准:

1. 关键回归测试输出只保留高信号内容

## P1. 深化消息与会话一致性验证

目标:

1. 不只验证 `/messages` 返回 201
2. 进一步验证 local-only 模式下的消息状态、会话更新、已读回执初始化是否真正落库

建议动作:

1. 扩展 `test/app.e2e-spec.ts`
2. 断言响应体中的 `message.status`
3. 断言 `message.extra.imDispatchMode === "local-only"`
4. 必要时补一条会话查询或仓储级验证

退出标准:

1. E2E 覆盖从“接口可调用”提升到“关键业务语义正确”

## P2. 继续排查前缀路由一致性

目标:

1. 确认不只是两个过滤器
2. 其他日志、中间件、审计装饰器、监控打点也应优先使用外部原始路由

建议动作:

1. 审计 `request.url` / `request.path` / `req.originalUrl` 的使用点
2. 重点关注:
   - 审计日志
   - 监控指标
   - 其他异常过滤器
   - 请求追踪中间件

退出标准:

1. `/im/v3/*` 与 `/admin/im/v3/*` 全链路路径语义一致

## P3. 持续高风险组件审计

优先模块:

1. `conversation`
2. `message`
3. `user/auth`
4. `gateway/ws`

关注维度:

1. 关闭阶段是否仍有 fire-and-forget 任务
2. 是否存在测试 mock 污染运行时路径
3. 是否仍有“接口成功但后台异步失败”的隐蔽语义缺口

退出标准:

1. 再产出一轮审计记录与新问题清单
