### BUG-001 健康检查丢失非 Error 异常信息

- 严重级别: High
- 状态: 已验证修复
- 位置:
  - `src/common/health/health-check.service.ts`
  - `src/common/health/health.controller.ts`
- 症状:
  - 依赖健康检查若抛出字符串等非 `Error` 对象，健康返回体中的错误信息可能为 `undefined`
- 根因:
  - 代码假设所有异常都是 `Error`
- 影响:
  - 生产故障时，健康检查信息不完整，影响定位
- 修复方案:
  - 将异常统一归一化为字符串
  - 去除危险的非空断言
  - 为依赖详情返回体补类型约束
- 验证:
  - 新增健康检查 service/controller 回归测试并通过

### BUG-002 任务调度器对 schedule 参数缺少强约束

- 严重级别: High
- 状态: 修复中
- 位置: `src/common/schedulers/task-scheduler.service.ts`
- 症状:
  - `cron` / `interval` / `timeout` / `once` 依赖 `!` 非空断言
  - 调用方若传入不完整参数，会在运行时以隐蔽方式失败
- 根因:
  - `ScheduleOptions` 使用宽松可选字段，但未在 `schedule` 入口做针对类型的参数校验
- 影响:
  - 错误在生产环境才暴露，且错误上下文不清晰
- 修复方案:
  - 入口校验任务类型对应的必要参数
  - 去除非空断言
  - 为非法输入补充单测
- 验证计划:
  - 定向 jest
  - `npx eslint src/common/schedulers/task-scheduler.service.ts`
  - `npm run lint:types`
  - `npm run lint`

### BUG-003 任务调度器 pause/resume 语义错误

- 严重级别: High
- 状态: 修复中
- 位置: `src/common/schedulers/task-scheduler.service.ts`
- 症状:
  - `pause()` 对非 `cron` 任务仅更新 `status`
  - `interval` / `timeout` / `once` 仍可能继续触发或丢失恢复语义
- 根因:
  - 任务状态与底层 timer 句柄没有统一建模
- 影响:
  - 运维认为任务已暂停，但实际执行行为不可靠
  - 一次性任务暂停后可能无法正确恢复
- 修复方案:
  - 为定时器生命周期建立统一追踪
  - 明确不同调度类型的 pause/resume 语义
  - 最少保证: 暂停后不执行，恢复后行为可预测
- 验证计划:
  - 使用 fake timers 编写暂停/恢复回归测试

### BUG-004 任务调度器可恢复重试定时器未被托管

- 严重级别: High
- 状态: 修复中
- 位置: `src/common/schedulers/task-scheduler.service.ts`
- 症状:
  - `recoverable` 失败后使用 `setTimeout` 发起重试，但没有纳入统一管理
- 根因:
  - 重试 timer 未进入 `timeouts`/统一资源清理模型
- 影响:
  - 模块销毁、任务取消后仍可能发生“幽灵执行”
  - 内存和生命周期不可控
- 修复方案:
  - 把重试 timer 纳入统一追踪与清理逻辑
  - 取消/销毁时同步清理
- 验证计划:
  - 新增取消后不再重试的回归测试
  - 新增 `onModuleDestroy` 后不再触发任务的回归测试

### BUG-005 启动日志存在终端编码兼容性风险

- 严重级别: Medium
- 状态: 已确认
- 位置:
  - `src/main.ts`
  - `src/app.module.ts`
- 症状:
  - 含装饰字符和中文输出，跨 shell/跨平台日志易出现乱码
- 根因:
  - 控制台输出依赖终端编码和字体，且未做编码适配
- 影响:
  - 首次部署和故障排查的日志可读性下降
- 修复方案:
  - 将关键启动日志收敛为 ASCII 安全输出
  - 把复杂 banner 变为可选、可配置输出
- 验证计划:
  - Windows PowerShell / Linux shell / CI 日志输出检查
