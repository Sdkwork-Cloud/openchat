# OpenChat Agent Memory 混合检索参数闭环设计

日期: 2026-04-06
范围: `apps/openchat`

## 背景

在 `agent memory` 模块中，混合检索本应允许上层调用方控制两个关键维度：

1. `type`
   - 指定记忆类别或作用域，避免不同类型记忆被错误混检
2. `threshold`
   - 指定语义相似度下限，避免低相关结果进入排序和返回集合

此前的问题不是接口不存在，而是参数在调用链上没有完整闭环，导致上层传入值在中途被丢失。

## 问题表现

调用链中的问题主要体现在三个层面：

1. Controller 接收到了 `type` 与 `threshold`
2. Interface 没有完整表达这两个参数的意图
3. Memory Manager 在执行 hybrid search 时没有稳定使用这两个值

最终结果是：

1. 业务层误以为自己在按类型过滤记忆，实际没有生效
2. 业务层误以为自己在控制召回阈值，实际结果集可能包含低相关噪声
3. 参数语义和实际执行路径发生偏离，接口契约不可信

## 设计目标

本轮收敛目标如下：

1. `type` 与 `threshold` 在 `controller -> interface -> manager` 全链路可见
2. 未传参数时保持兼容，不破坏旧调用方
3. 参数生效点明确，可通过单元测试验证

## 设计方案

### 1. 参数透传

要求：

1. Controller 层不得只接收不透传
2. Interface 层不得遗漏参数定义
3. Manager 层不得忽略上游参数

设计含义：

- 所有公开检索入口对 `type` 和 `threshold` 的处理必须保持一致
- 任何中间层都不能再把这两个参数当成“未来再接”的占位字段

### 2. `type` 的语义

`type` 用于限制记忆检索的作用域，典型场景包括：

1. 只查询某一类用户偏好
2. 只查询某一类历史上下文
3. 避免系统记忆、会话记忆、长期记忆互相污染

实现原则：

1. `type` 缺省时保持原有宽松检索行为
2. `type` 传入时仅对目标类别进行检索或过滤

### 3. `threshold` 的语义

`threshold` 用于控制混合检索中相似度过滤的下限。

实现原则：

1. `threshold` 缺省时使用现有默认行为
2. `threshold` 传入时在候选结果进入最终集合前生效
3. 不允许上层显式传入阈值但执行链完全忽略

### 4. 兼容性原则

为避免破坏现有调用方：

1. 参数仍保持可选
2. 不修改已有成功路径的默认结果结构
3. 仅在调用方显式传参时改变检索约束

## 验证策略

本轮通过以下测试验证参数闭环已经成立：

1. `src/modules/agent/memory/memory.controller.spec.ts`
2. `src/modules/agent/memory/memory-manager.service.spec.ts`

验证重点：

1. Controller 是否将 `type` 与 `threshold` 正确传递给下层
2. Manager 是否在 hybrid search 中实际应用这两个参数

## 收益

参数闭环完成后，`agent memory` 的收益主要有三点：

1. 接口契约可信
   - 上层传入的检索约束真正进入执行链
2. 结果质量更稳定
   - 低相关结果可被阈值剔除
3. 类型边界更清晰
   - 不同记忆域之间的污染风险下降

## 后续建议

1. 为 `threshold` 增加统一的边界校验与默认值文档
2. 在 API 文档中明确 `type` 的合法枚举范围
3. 若未来引入多向量源或 rerank 流程，继续保持 `threshold` 语义在全链路一致
