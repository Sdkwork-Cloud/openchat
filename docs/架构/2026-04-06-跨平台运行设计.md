# OpenChat 跨平台运行设计

日期: 2026-04-06
目标平台: Windows、Linux、macOS

## 1. 设计目标

跨平台不是“脚本能执行”，而是以下链路都要稳定一致:

1. 安装依赖
2. 初始化环境变量
3. 初始化数据库
4. 启动开发环境
5. 执行测试
6. 构建产物
7. 执行基础运维操作

## 2. 当前基础

仓库已经同时提供:

- `npm scripts`
- `node scripts/*.cjs`
- PowerShell 脚本: `*.ps1`
- Shell 脚本: `*.sh`
- Docker / Compose 相关脚本

这是优势，但也意味着必须定义“主路径”和“平台适配路径”。

## 3. 主路径设计

推荐主路径:

1. 应用开发/测试/构建
   - 统一使用 `npm run ...`

2. 环境注入与工具逻辑
   - 统一下沉到 `node scripts/*.cjs` 或 `*.js`

3. OS 专属脚本
   - `*.ps1` 和 `*.sh` 只负责包装和平台差异处理

这样做的原因:

- Node 脚本比 shell 更容易保持跨平台一致性
- CI 与本地开发可以复用同一套逻辑
- PowerShell/bash 差异不会污染主业务流程

## 4. 当前确认的跨平台风险

### 风险 1: 启动日志编码兼容性不足

位置:

- `src/main.ts`
- `src/app.module.ts`

问题:

- 使用大量装饰字符和中文输出
- 在 Windows PowerShell、不同 code page、无中文字体环境或 CI 终端下可能出现乱码

方案:

- 核心诊断日志改为 ASCII 安全输出
- banner 输出改为可配置开关

### 风险 2: 运维脚本存在双轨实现

问题:

- 同一类操作通常同时存在 `.ps1` 与 `.sh`
- 若逻辑分叉，长期容易出现功能漂移

方案:

- 核心逻辑收敛到 Node CLI
- shell/powershell 只负责参数透传、系统检查和用户友好入口

### 风险 3: 路径、换行、编码未形成统一规范

问题:

- Windows 与 Unix 平台在路径分隔符、换行符、执行权限上存在天然差异

方案:

- 统一使用 Node 的 path API 处理路径
- 文本文件统一 UTF-8
- 对 shell 脚本、配置文件建立换行和执行权限规范

## 5. 跨平台运行标准

### 开发标准

- `npm ci`
- `npm run start:dev`
- `npm run lint`
- `npm run lint:types`
- `npm test`

### 数据库标准

- 首选 `npm run db:init:*` / `npm run db:patch:*`
- `ps1` / `sh` 仅作为平台友好包装

### 生产标准

- 优先容器化部署
- 本地脚本用于运维辅助，而不是唯一生产入口

## 6. 兼容性测试矩阵

至少需要覆盖:

1. Windows PowerShell
   - 安装
   - 开发启动
   - lint/types/test/build
   - db init / db patch

2. Linux bash
   - 安装
   - 开发启动
   - lint/types/test/build
   - db init / db patch

3. macOS zsh
   - 安装
   - 开发启动
   - lint/types/test/build

## 7. 落地顺序

1. 修复启动日志编码风险
2. 固化 `npm` + Node CLI 为主入口
3. 对脚本职责做目录级说明
4. 补跨平台验证清单与结果记录
