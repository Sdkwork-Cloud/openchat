# OpenChat 流式导出单次消费与 JSON 边界设计

日期: 2026-04-06
范围: `apps/openchat`

## 背景

`src/common/services/export.service.ts` 和 `src/common/services/data-export.service.ts` 都提供了流式导出能力，但原实现存在两个结构性问题：

1. 把 `AsyncIterable` 当成可以重复遍历的数据源
2. 对 JSON 流式写出缺少明确的数组边界管理

这两个问题都会在数据量变大或数据源变成一次性流之后暴露为真实业务故障。

## 设计原则

### 1. `AsyncIterable` 只能按单次消费模型设计

规则:

- 不允许为了统计总数先完整遍历一次，再进行第二次业务遍历
- 过滤、转换、字段裁剪、批次写出必须在同一轮消费中完成

原因:

1. 数据源可能来自数据库 cursor、远程分页流或消息总线
2. 这些来源天然只保证单次读取，不保证可回放
3. 双重遍历会让“统计”本身变成破坏性操作

### 2. JSON 流式写出必须显式管理容器边界

规则:

- 首批负责写出 `[` 与首批元素
- 中间批次只负责写出 `,` 与元素内容
- 结束阶段统一写出 `]`
- 空结果必须显式写出 `[]`

原因:

1. `JSON.stringify(batch)` 会产生一个局部数组，不适合直接多批拼接
2. 只有把“容器生命周期”从“批次内容”中拆出来，才能保证最终文件始终可解析

### 3. 流式与非流式导出应尽量保持语义一致

规则:

- 字段投影
- 字段转换
- 编码行为
- 目录准备

以上语义在普通导出和流式导出之间应保持一致，不能因为走了流式路径就变成另一套结果。

## 本轮落地规则

### 规则 A: `ExportService.exportStream()`

- 只消费一次 `dataStream`
- 在单轮遍历中完成过滤、转换、字段裁剪
- 分批写出时处理写流背压
- 导出前确保目录存在

### 规则 B: `DataExportService.exportStream()`

- 使用 `isFirstBatch` 明确区分首批与后续批次
- JSON 首批写出数组起始符
- 完成后统一关闭数组
- 空流时输出 `[]`

## 收益

1. 避免一次性流被预统计逻辑吞空
2. 保证 JSON 流式文件始终可被标准解析器读取
3. 降低大批量导出时的内存压力和写流阻塞风险
4. 让后续 E2E 和接口联调更容易验证导出行为

## 后续建议

1. 为 `ExportService` 的 JSON 流式格式补充契约定义，明确是数组 JSON 还是 JSONL
2. 为空结果、过滤后零结果、超大批次和异常中断场景补充更多测试
3. 在导出进度模型中增加“未知总量”语义，避免用 `percentage` 表示不可预估总数的流式任务
