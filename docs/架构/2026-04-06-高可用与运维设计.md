# OpenChat 高可用与运维设计

日期: 2026-04-06
目标: 将当前已有的 health、metrics、queue、cache、db、IM provider 能力收敛为可运维、可恢复、可扩展的运行体系。

## 1. 运维设计目标

必须具备以下能力:

1. 故障可发现
2. 故障可定位
3. 故障可隔离
4. 故障可恢复
5. 发布可回滚

## 2. 当前可用基础

仓库中已存在的基础能力:

- 健康检查模块
- 指标与监控服务
- Redis/队列模块
- 数据库配置和初始化脚本
- WebSocket/Gateway 与 IM provider 集成

说明:

- 这些基础已经足够支撑高可用设计
- 当前不足在于实现质量和运维策略尚未完全收敛

## 3. 推荐部署拓扑

### 应用层

- OpenChat Server 多副本部署
- 接入层使用负载均衡或 Ingress
- WebSocket 连接需要结合 Redis/共享状态做多实例协同

### 数据层

- PostgreSQL 主从或托管高可用方案
- Redis 哨兵/集群或托管高可用方案

### 消息与外部依赖

- BullMQ 使用独立 Redis 资源或与缓存隔离策略
- WuKongIM / RTC / AI provider 作为外部依赖，必须监控可用性和超时

## 4. 健康检查设计

健康检查应拆分为三类:

1. `startup`
   - 进程是否启动成功
   - 配置是否可加载

2. `readiness`
   - 数据库、Redis、队列、IM provider 是否可提供服务

3. `liveness`
   - 进程是否卡死
   - 事件循环是否异常阻塞

当前行动:

- 已先修复健康检查错误归一化问题，避免依赖异常信息丢失

## 5. 可观测性设计

### 指标

至少应包含:

- HTTP 请求量、错误率、延迟
- WebSocket 在线连接数
- Redis/DB/队列依赖状态
- 消息发送成功率与失败率
- 定时任务执行次数、失败次数、平均耗时

### 日志

要求:

- 结构化
- 可按 traceId / userId / taskId / messageId 检索
- 错误日志不丢根因信息

当前问题:

- 启动日志样式化较重，不利于跨平台诊断
- 公共层错误处理仍需持续收敛

## 6. 容错与降级设计

### 数据库异常

- 健康检查暴露为 not ready
- 非关键后台任务降级或延迟执行
- 读写异常必须记录可定位上下文

### Redis 异常

- 限流、缓存、会话、队列能力可能受影响
- 必须明确哪些链路可降级，哪些链路直接失败

### IM/RTC/AI 外部依赖异常

- 统一 provider 层超时、重试和错误映射
- 避免业务层直接散落外部 SDK 调用

### 调度与后台任务异常

- 任务暂停、恢复、取消、销毁必须语义一致
- 不能在应用停止后继续发生幽灵执行

## 7. 运维落地优先级

1. 修复基础健康检查与调度器生命周期问题
2. 统一关键日志和错误归一化
3. 建立 readiness/liveness/startup 的明确边界
4. 补部署文档与故障处理流程
5. 再推进更细粒度的容量治理和弹性扩缩容

## 8. 最终目标

最终交付标准不是“代码能启动”，而是:

- 在多操作系统环境可稳定运行
- 在关键依赖故障时能正确暴露状态
- 在任务失败、依赖超时、节点重启时行为可预测
- 在运维和排障时日志、指标和健康信息足够可信
