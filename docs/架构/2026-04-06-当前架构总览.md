# OpenChat 当前架构总览

日期: 2026-04-06
适用范围: `apps/openchat`

## 1. 架构分层

当前应用本质上是“单体业务服务 + 实时通信网关 + 外部 IM/RTC/AI 集成”的组合架构。

分层如下:

1. 接入层
   - HTTP API
   - WebSocket Gateway
   - Webhook 入口

2. 业务层
   - `user`
   - `message`
   - `group`
   - `conversation`
   - `friend`
   - `presence`
   - `rtc`
   - `agent`
   - `iot`
   - `ai-bot`

3. 平台能力层
   - `auth`
   - `cache`
   - `queue`
   - `metrics`
   - `health`
   - `events`
   - `redis`
   - `extensions`

4. 外部依赖层
   - PostgreSQL
   - Redis
   - BullMQ
   - WuKongIM / IM provider
   - RTC 厂商能力
   - OpenAI / 其他 AI provider

## 2. 现状判断

当前架构不是“设计错误”，而是“公共层收敛不足”。主要问题体现在:

- 平台能力层承担了大量全局责任，但边界约束不够严格
- 部分公共服务存在类型安全不足、生命周期管理不完整、错误处理不统一的问题
- 文档中已有总体架构描述，但缺少“当前实现状态”和“治理方向”的说明

## 3. 关键调用路径

### 消息主链路

客户端 -> HTTP/WebSocket -> Message/Conversation/Group 等业务模块 -> IM Provider/WuKongIM -> 回执/推送

关键要求:

- 消息状态一致性
- 重试与幂等
- 读写链路监控
- 异常可追踪

### 登录与鉴权链路

客户端 -> Auth Guard / Strategy -> User/Auth Service -> Redis / JWT -> API/Gateway

关键要求:

- 多认证源一致性
- Token 生命周期可控
- 网关侧与 API 侧权限判断一致

### RTC / IoT / Agent 扩展链路

外部服务事件 -> 扩展/控制器/网关 -> 业务服务 -> 队列/缓存/数据库

关键要求:

- 外部依赖故障时可降级
- 超时、重试、熔断边界清晰

## 4. 当前架构痛点

### 4.1 公共层承担太多运行时复杂性

表现:

- 健康检查、调度器、日志、异常、缓存、队列、监控都集中在 `src/common`
- 一旦公共层边界不严，会影响大量业务模块

治理原则:

- 优先修公共层真实缺陷
- 对公共层建立更严格的类型和测试约束

### 4.2 调度、重试、生命周期模型有待统一

表现:

- 不同服务中散落大量 `setTimeout` / `setInterval`
- 停止、销毁、暂停、恢复语义未统一

治理原则:

- 把定时器从“临时实现”收敛为“可管理资源”
- 明确取消、暂停、销毁后的行为保证

### 4.3 跨平台启动与运行策略需要明确主路径

表现:

- 仓库同时存在 `npm`、`cjs`、`.ps1`、`.sh`、Docker 多入口
- 若不收敛主入口，容易导致文档与真实运行方式不一致

治理原则:

- 优先以 `npm` + `node scripts/*.cjs` 作为统一入口
- 平台脚本作为适配层，而不是主逻辑层

## 5. 推荐收敛目标

### 5.1 结构目标

- HTTP/WebSocket/Webhook 只负责接入
- 业务模块只负责领域逻辑
- 公共层只负责平台能力，不侵入业务细节
- 外部依赖通过 provider/adapter 形式接入

### 5.2 工程目标

- 类型边界更严格
- 错误归一化统一
- 定时器/任务生命周期可托管
- 所有关键修复必须附带回归测试

### 5.3 运维目标

- 健康检查可反映真实依赖状态
- 指标与日志可支持故障定位
- 构建、启动、数据库初始化路径跨平台可执行

## 6. 当前建议

最优路径不是立即拆微服务，而是先完成以下治理:

1. 收敛公共层缺陷
2. 建立跨平台主运行路径
3. 把 health / metrics / queue / scheduler 的行为做实
4. 再评估消息、RTC、IoT、Agent 的进一步拆分
