# OpenChat 公共错误归一化与质量闸门设计

日期: 2026-04-06
范围: `apps/openchat`

## 背景

本轮修复中，阻塞构建的核心问题不是业务逻辑本身，而是大量公共边界没有统一处理 `unknown` 异常，导致:

- 编译器无法确认 `.message`
- `Error` 类型赋值断裂
- 不同模块的日志和异常处理策略不一致
- 后续 `preserve-caught-error` 与可观测性治理难以统一

因此需要建立公共错误归一化能力，并把它作为质量闸门的一部分。

## 设计目标

1. 所有模块都能安全处理 `unknown` 异常
2. 统一日志错误消息提取逻辑
3. 在需要 `Error` 实例时，统一完成包装
4. 在读取第三方错误结构时，避免不安全的对象访问
5. 配合 `lint:types` / `build` / `lint` 形成可持续验证闭环

## 设计方案

新增公共工具:

- `getErrorMessage(error: unknown): string`
- `toError(error: unknown): Error`
- `asRecord(value: unknown): Record<string, unknown> | undefined`

使用原则:

- 只要 `catch` 到的是 `unknown`，日志输出统一走 `getErrorMessage`
- 只要目标字段要求 `Error`，统一走 `toError`
- 只要需要读取第三方返回对象的嵌套结构，统一走 `asRecord`

## 质量闸门

本设计要求每一轮改动至少通过以下闸门:

1. `npm run lint:types`
2. `npm run build`
3. 受影响模块的定向测试
4. `npm run lint`

策略说明:

- `lint:types` 和 `build` 属于硬闸门，必须持续通过
- `lint` 当前因历史债务过多仍未完全通过，但每轮都必须压低 error 总数
- 任何新的错误处理代码，禁止再引入直接访问 `unknown.message` 的写法

## 架构收益

引入该设计后，收益体现在:

- 公共层错误处理逻辑统一
- 类型系统约束更强
- 业务模块不必重复发明错误提取逻辑
- 后续可以继续向“错误分类、错误码映射、cause 链保真、监控归因”演进

## 后续演进

下一步建议:

1. 在 `src/common` 下继续统一错误 cause 处理策略
2. 将更多散落的 `error instanceof Error ? ...` 收敛到公共工具
3. 在审计日志、健康检查、事件总线中统一错误上下文结构
4. 逐步推动 lint warning 燃尽，避免新的类型松散边界再次扩散
