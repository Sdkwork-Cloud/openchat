# OpenChat 公共核心运行时兼容性设计说明

日期: 2026-04-06
范围: `apps/openchat`

## 背景

`src/common/types`、`src/common/utils`、`src/common/websocket` 是整个 OpenChat 应用系统的公共核心层。这些文件会被服务层、扩展层和模块层广泛依赖，因此这里的代码风格和运行时假设会直接影响跨环境可运行性。

本轮收口中识别出两个典型问题:

1. 对全局 `crypto` 标识符存在隐式依赖
2. 条件类型与公共辅助类型中存在只为“写得出来”而留下的无意义符号

## 设计原则

### 1. 公共工具不得依赖模糊的全局环境假设

规则:

- 在需要使用 Web Crypto API 时，统一显式访问 `globalThis.crypto`
- 不假设裸 `crypto` 标识符在所有运行环境和静态分析器中都存在

原因:

1. Node.js、浏览器、测试环境和静态分析器对全局符号的识别并不完全一致
2. `globalThis` 是跨运行时最明确的访问边界
3. 这样可以减少“运行时可用，但 lint 失败”或“某些环境可用，某些环境不可用”的隐性耦合

### 2. 公共类型工具应避免无意义占位符

规则:

- 条件类型中的 `infer` 只在真正需要时引入名字
- 对不需要命名的返回值推断，直接使用 `unknown`

原因:

1. 无意义的类型占位符会触发静态规则噪音
2. 这些噪音会掩盖真正有价值的问题
3. 公共类型层越干净，下游模块越容易收敛

### 3. 公共网关基类应只保留实际参与行为的参数和状态

规则:

- `afterInit`、速率限制、连接管理等基础能力中，不保留未参与逻辑的入参或局部变量
- 需要保留框架签名时，用 `_param` 明确表达占位语义

原因:

1. 公共基类会被大量继承，噪音会成倍放大
2. 无意义状态会误导后续开发者，以为存在未完成的逻辑依赖

## 本轮落地规则

1. `src/common/utils/index.ts`
   - `generateUuid()` 与 `hash()` 改为使用 `globalThis.crypto`
2. `src/common/types/index.ts`
   - `UnionToTuple` 中的无意义 `infer _` 改为 `unknown`
3. `src/common/websocket/base-websocket.gateway.ts`
   - 删除未使用的 `MessageBody` import
   - 对未使用初始化参数显式使用 `_server`
   - 去除无效局部变量

## 收益

1. 公共核心层对 Node.js 与浏览器风格运行时更友好
2. 静态分析与运行时行为更一致
3. 下游模块在继续收口时能更容易识别真实问题
4. 对不同操作系统和不同执行入口的兼容性更稳

## 后续建议

1. 继续排查 `extensions/*` 与网关入口中是否存在类似的隐式运行时依赖
2. 为公共工具补充更系统的单元测试，尤其是 UUID、哈希和批处理辅助函数
3. 在后续跨平台交付阶段，增加一轮 Linux/Windows 路径与运行时差异专项验证
