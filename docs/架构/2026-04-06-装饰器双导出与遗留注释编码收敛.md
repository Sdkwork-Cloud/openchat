# OpenChat 装饰器双导出与遗留注释编码收敛设计

日期: 2026-04-06
范围: `apps/openchat`

## 背景

在 `src/common/decorators/user-context.decorator.ts` 中，参数装饰器与其对应的类型接口需要保持同名导出，外部调用方才可以同时使用:

- `@PaginationParams()` 这类 decorator
- `PaginationParams` 这类类型

同时，历史代码中还存在少量损坏注释，既影响 `eslint`，又不适合在当前轮次做全文件编码迁移。

## 设计原则

### 1. 类型与值允许同名导出，但 ESLint 例外必须局部化

TypeScript 允许类型空间和值空间同名共存，因此以下模式是合法且必要的:

```ts
export interface PaginationParams {
  page: number;
}

// eslint-disable-next-line no-redeclare
export const PaginationParams = createParamDecorator(/* ... */);
```

这样做的收益:

1. 对外 API 最直观
2. 不破坏已有调用点
3. 避免“重命名后再导出”导致声明发射引用到私有符号

约束:

- 只允许在确有类型/值双导出需求的声明上使用 `no-redeclare` 局部例外
- 禁止扩大到整文件级别

### 2. 遗留编码污染先做受控隔离，再做统一迁移

对于损坏注释引发的 `no-irregular-whitespace`:

- 当前轮次不做整文件编码转换
- 只在受影响注释块附近做最小范围的 `eslint-disable/enable`
- 同时补充一条可读的 ASCII 注释说明方法语义

收益:

1. 不会误伤业务代码
2. 不会把一次小修复放大为大面积文本变更
3. 便于后续统一做编码治理

## 已落地规则

### 规则 A: Nest 参数装饰器导出模式

适用范围:

- `src/common/decorators/*`

规则:

1. 接口保留原有公开类型名
2. decorator 常量与接口同名导出
3. 仅对该行添加 `// eslint-disable-next-line no-redeclare`

### 规则 B: 遗留乱码注释处理模式

适用范围:

- 仅限历史损坏注释且无法安全自动替换的文件

规则:

1. 使用局部 `eslint-disable no-irregular-whitespace`
2. 紧随其后补充一条新的可读注释
3. 不把 suppression 扩大到整文件

## 后续建议

1. 在后续独立轮次中评估统一文本编码治理，避免继续保留损坏注释
2. 在公共 decorator 目录内形成统一示例，减少未来重复踩坑
3. 若再出现类型/值同名导出需求，优先复用本设计，而不是再做“重命名后别名导出”
